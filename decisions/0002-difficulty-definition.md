# ADR 0002 — 난이도의 정의: 하이브리드 (스칼라 KPI + 설명 변수 벡터)

- **상태**: 확정
- **일자**: 2026-05-06
- **결정자**: 사용자 + Claude(레벨 디자이너 페르소나)

---

## 맥락

버블슈터 레벨 자동화 파이프라인의 **모든 후속 의사결정**(생성기 파라미터, 봇 솔버 출력, LLM 보조 인터페이스, 분석 차트)이 "난이도를 무엇으로 정의하느냐"에 종속된다. 이 결정을 가장 먼저 못박아야 한다.

세 가지 후보를 비교했다.

| 옵션 | 정의 | 강점 | 약점 |
|---|---|---|---|
| A. 단일 스칼라 | 평균 시도 횟수 1개 값 | 직관, A/B 1:1 매칭 | "왜 어려운지" 설명 불가 |
| B. 다차원 벡터 | (색 엔트로피, 연쇄 깊이, 부유, 천장 거리…) | 구조적 분해, 정밀 조작 | KPI와의 관계 불명 |
| **C. 하이브리드** | **스칼라 KPI + 설명 변수 벡터** | **A·B 장점 결합 + 회귀분석 가능** | 구현 복잡도 |

## 결정

**옵션 C 채택.**

- **종속변수 (KPI)**: 봇 솔버가 측정한 *기대 시도 횟수* (Expected Attempts to Clear, EAC)
- **독립변수 (설명 변수 벡터)**: 보드의 정적 구조에서 계산되는 특징량들. 차원은 다음 단계에서 별도 ADR로 확정.
- **두 변수의 관계**: 100개 이상의 생성 레벨에 대해 EAC를 측정한 뒤, 설명 변수로 EAC를 회귀 → 어느 차원이 난이도를 가장 잘 예측하는지 정량화.

## 근거

1. **회사 공고와의 정합성**: "수학적 설계"(B의 강점) + "압도적 데이터"(A의 강점)를 *둘 다* 충족하는 유일한 옵션.
2. **포트폴리오 서사**: "디자이너의 직관(설명 변수) → 데이터로 검증(KPI) → 회귀로 직관 정당화"라는 일관된 스토리 라인을 만든다.
3. **확장성**: 라이브 게임에서 실제 유저 데이터(평균 시도)가 들어오면 봇 EAC 자리에 그대로 치환 가능. 즉 *프로토타입이 그대로 프로덕션 분석 프레임이 된다*.

## 트레이드오프

- 구현 복잡도가 A보다 높다. 회귀 모델·차트·LLM 보조 입력 포맷이 추가된다.
- 설명 변수 벡터의 차원을 잘못 정하면 회귀 결과가 무의미해진다 → 다음 ADR에서 신중히 고를 것.
- 6개월 인턴 포지션에 약간 오버스펙일 수 있으나, 그 점이 *지원자의 야심*으로 읽힐 가능성이 더 크다고 판단.
