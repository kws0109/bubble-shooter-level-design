# AI 협업으로 만드는 버블슈터 레벨 디자인 파이프라인

> **버블슈터 레벨 디자인 포트폴리오**
> 결과물뿐 아니라 *AI와 함께 사고하고 결정한 과정* 자체를 보여줍니다.

---

## 0. 한 줄 요약

> 버블슈터 레벨을 **절차적으로 생성**하고, **봇 시뮬레이션으로 난이도를 계측**하며, **LLM을 디자인 보조로 활용**해 데이터 기반으로 튜닝하는 파이프라인을 구축합니다.

## 1. 왜 이 포트폴리오인가

**문제**: "AI를 활용해 퍼즐 레벨을 설계한다"는 직무를 어떻게 증명할 것인가?

대부분의 지원자는 두 가지 중 하나를 한다:
- (A) 완성된 레벨 산출물만 제출 → AI를 *어떻게* 썼는지 안 보임
- (B) "ChatGPT에 시켰어요" 식의 얕은 활용 → 디자이너로서의 사고력이 안 보임

이 포트폴리오는 다른 길을 택했다. **과정 자체를 결과물로 만든다.**
- 생성·계측·튜닝의 파이프라인을 직접 설계
- 모든 의사결정에 *이유와 트레이드오프*를 기록
- 실패와 재시도까지 남겨 신뢰도 확보

회사 공고가 강조한 "수학적 설계", "압도적 데이터", "가설과 실험"이라는 언어를 제 작업물로 직접 번역하는 것이 목표다.

## 2. 왜 버블슈터인가

| 후보 | 이유 | 강점 | 약점 | 결과 |
|---|---|---|---|---|
| Word Cookies (워드 퍼즐) | 회사 1순위 IP | 한국어 차별화 가능, LLM 친화 | 단어 사전 의존도 높음 | 후순위 |

## 3. 디자인 가설

### 3.1. 난이도의 정의 — 하이브리드 모델

이 프로젝트의 첫 번째 핵심 가설은 다음과 같다.

> *버블슈터의 체감 난이도는 단일 숫자(평균 시도 횟수)로 측정할 수 있지만, **왜 어려운지는 보드의 구조적 특징 벡터로만 설명된다.** 두 변수의 관계는 데이터로 회귀해야 한다.*

이 가설을 따라 난이도를 두 층으로 나눈다.

- **종속변수 (KPI)**: **EAC** (Expected Attempts to Clear) — 봇이 1000번 플레이한 평균 시도 횟수
- **독립변수 (설명 변수 벡터)**: 보드의 정적 구조 특징 (색 엔트로피, 연쇄 깊이, 부유 가능성, 천장 거리 등 — ADR 0003에서 확정)
- **두 변수의 관계**: 100개 이상의 레벨에서 측정한 (벡터 → EAC) 데이터로 회귀 → 어느 차원이 난이도를 가장 잘 예측하는지 정량화

**왜 이 분리가 중요한가**: 라이브 게임의 실제 유저 데이터가 들어오면 EAC 자리에 *진짜 평균 시도 횟수*를 그대로 끼워넣을 수 있다. 즉 이 프로토타입은 *프로덕션 분석 프레임의 축소판*이다.

상세: [ADR 0002 — 난이도의 정의](decisions/0002-difficulty-definition.md), [ADR 0003 — 6차원 벡터](decisions/0003-feature-vector.md)

#### 봇 측정의 디자인 — weak/medium/strong 트리플렛
EAC 단일 봇은 *어려운 레벨에서 측정 불가*(못 끝내면 통계 제외)라는 함정이 있다. 셋업 휴리스틱 강도가 다른 봇 셋을 두고, *클리어율 spread*를 신호로 쓴다. StrongBot은 룩어헤드를 top-K로 트리밍해 ~3배 가속됐다.

### 3.2. 남은 핵심 질문

- 설명 변수 벡터의 차원을 정확히 몇 개, 어떻게 잡을 것인가? (다음 ADR)
- "재미"의 flow 구간을 EAC 분포로 어떻게 정의할 것인가? (예: EAC 3~7회 사이가 sweet spot)
- 봇의 휴리스틱이 *유저보다 너무 잘하면* EAC가 과소평가된다 — 봇 강도를 어떻게 보정할 것인가?

## 4. AI 활용의 3층위

이 프로젝트에서 AI는 한 번이 아니라 **세 가지 다른 역할**로 쓰인다.

### Layer 1. 절차적 생성기 (`src/bubble/generator.py`)
- **Stage 1 (시드+성장)**: K개 시드를 거리 제약으로 배치, 각 색을 인접 셀로 확률적 성장. 디자이너의 *형태적 의도*가 보드에 들어간다.
- **Stage 2 ((1+1) 진화)**: 6차원 목표 벡터와의 가중 거리를 적합도로 변형 수락/기각. 첫 실험에서 적합도 626 → 0.5 (1000× 감소).
- 결정론적 시드 → 재현 가능성 확보.
- 결정 근거: [ADR 0004](decisions/0004-generator-strategy.md)

### Layer 2. 봇 플레이어 (`src/bubble/solver.py`)
- WeakBot / MediumBot / StrongBot 셋이 같은 레벨을 N회 플레이 → 클리어율과 EAC 측정.
- "체감 난이도"를 정량으로 환산하는 *난이도 계측기*

- "체감 난이도"를 정량으로 환산하는 *난이도 계측기*. EAC와 clear_rate 둘 다 종속변수로 활용.

### Layer 3. LLM 디자인 보조 (`src/bubble/ai_designer.py`)
- 입력: 보드 + 현재 6차원 벡터 + 목표 벡터 (+ 회귀 분석 컨텍스트)
- 출력: `{"rationale": "...", "edits": [{"row","col","color"}, ...]}`. *구조화된 패치*만 받음 (자유 생성 금지)
- 우리 솔버로 패치 적용 → 재측정 → 수렴 루프 (수락/기각)
- API 키 없으면 *모킹 폴백*으로 데모 가능. 첫 실험에서 거리 0.224 → 0.025 수렴 + 1회 기각(검증 루프 정상)

이 3층 구조가 이 포트폴리오의 핵심 차별점이다. *생성을 LLM에 통째로 맡기지 않는다.*

## 6. 실험 결과 (n = 100)

### 6.1. 봇 트리플렛 분포

| 봇 | 평균 클리어율 | std | 범위 | 평균 EAC_all |
|---|---|---|---|---|
| weak   | 57.7% | 0.13 | 0.23 – 0.87 | — |
| medium | 63.6% | 0.13 | 0.30 – 0.92 | — |
| strong | **88.9%** | 0.09 | 0.57 – 1.00 | 24.4 (std 5.8) |

발견:
- **weak/medium은 좋은 spread** (std 0.13). 회귀 종속변수로 적합.
- **StrongBot은 천장 효과**. 평균 89%, 1.00 도달 다수. 룩어헤드의 강함이 *난이도 측정의 유효 범위를 좁힘*.
- **EAC_all 7~37**: 시도 횟수 신호는 클리어율보다 *동적 범위가 넓다*. 어려운 레벨 변별에 더 유리.

### 6.2. 회귀 결과 — 솔직한 보고

15레벨 파일럿에서 R² ≈ 0.85였던 모델이 100레벨에서 **R² 0.22~0.29**로 떨어졌다.

| 종속변수 | R² (n=100) |
|---|---|
| weak_clear_rate | 0.223 |
| medium_clear_rate | 0.287 |
| strong_clear_rate | 0.239 |
| strong_eac_all | 0.287 |

표본을 늘렸을 때 R²가 떨어진 *원인을 추적했다*:

#### 발견 1 — 다중공선성: `corr(density, shot_pressure) = −0.976`
ADR 0003에서 두 차원이 *서로 독립*이라 가정했지만, **`shots_remaining`이 14로 고정된 셋업에서 `shot_pressure ≡ 14 / (density × cells)` ≈ density의 역수**가 됐다. 결과:
- density 계수 −0.142 + shot_pressure 계수 −0.138 (clear_rate 회귀)
- 둘이 같은 신호를 *나눠 들고* 있어 회귀가 흔들림.
- `shot_pressure` 제거 후 재회귀하면 density 계수도 0 근처로 무너짐 → 둘 합쳐 신호 1개를 표현하던 것.
- **정정**: ADR 0005에서 다음 이터레이션의 차원 제거를 결정.

#### 발견 2 — 단일 가장 안정된 예측 변수: `avg_cluster_size`
모든 종속변수에서 부호가 일관되고(+0.04~0.06 / EAC −2.77), 다중공선성과 무관하게 살아남음.
- 클러스터가 클수록 **클리어율 ↑, 시도 ↓** — 디자이너 직관과 일치 ("큰 같은색 무리는 한 발에 깨진다").

#### 발견 3 — 6차원으로 설명되지 않는 ~75% 분산
나머지를 분해하면:
- 봇이 받는 *색 수열*의 RNG (calibration_runs=60으로 노이즈 잔존)
- 6차원이 캡처 못하는 구조적 미시 특징 (예: 천장에서 가장 약한 다리 1개)
- 클리어 가능성의 *비선형 효과* (선형 회귀 한계)

이건 *모델이 부족한 것*이 아니라 *6차원만으로는 충분치 않다*는 정직한 신호다. 다음 이터레이션의 후보:
- shots_remaining을 변량화 → shot_pressure가 진짜 독립 차원이 되는지 재검증
- 2차 항(density × avg_cluster_size 등) 추가
- calibration_runs ↑로 봇 노이즈 감소

### 6.3. LLM 디자인 보조 수렴 (모킹 모드)

샘플 레벨 L000에 entropy +0.2, density +0.1을 요청 → 4라운드에서 가중 거리 **0.224 → 0.025로 수렴**, 1회 *기각*으로 검증 루프가 살아있음을 확인. Anthropic API 키 주입 시 동일 인터페이스로 LLM 호출.

### 6.4. 시각화

브라우저로 `viewer/index.html`을 열면 100레벨을 페이징하며 보드·피처·봇 통계를 한눈에 본다.

## 6.5. Iteration 2 — 라이브 게임 메커닉 도입

Iteration 1이 *난이도 측정 프레임*을 검증했다면, Iteration 2는 그 프레임 위에 **실제 라이브 게임 메커닉**을 얹는다. 디자이너 피드백 5개를 받아 시스템을 진화시켰다 — 라이브 운영의 표준 패턴.

### 변경 사항 (ADR 0006)
1. **레벨 진행에 따른 색 도입**: L1-10 = 2~3색, L11-20 = 3~4색, …, L41-50 = 6~7색, L51+ 고정.
2. **2색 큐 슈터**(Bubble Pop Origin 표준): 매 턴 두 색 중 하나를 골라 발사. 봇 인터페이스가 `choose_shot(board, queue, rng)` → `(row, col, qi)` 로 변경.
3. **좌우 대칭 보드**: Stage 1에 미러 시드, Stage 2에 70% 짝 변형. 시각적 대칭을 유지하되 살아있는 느낌.
4. **발사 수 보정**: `shots = 12 + 2 × (colors - 2)` — 색이 늘면 발사 수도 비례 증가.
5. 클리어 조건·부유 버블 자동 삭제는 Iteration 1에서 이미 구현됨.

### 결과 (n = 100, 두 차례 배치)

각 단계의 산출물이 보존되어 *직접 비교* 가능:
- `levels_iter1/`, `reports_iter1/` — iter1 (대칭 없음)
- `levels_iter2_soft/`, `reports_iter2_soft/` — iter2 (soft symmetry, paired_rate=0.7)
- `levels/`, `reports/` — iter2 strict (모든 placement가 mirror, 좌우 60% / 상하 40%)

#### Soft → Strict 전환의 동기

soft 버전의 보드는 *좌우에 같은 색이 비슷한 위치*에 있긴 했지만, 30% 비대칭 변형이 600세대 누적되면서 *진짜 패턴으로 안 보임*. 디자이너 피드백을 받아 **모든 placement가 미러 셋에 동시 적용**되도록 재설계 (Stage 1 성장 + Stage 2 변형 둘 다).

#### 세 배치의 분포 비교

| 봇 | iter1 mean (std) | iter2 soft (std) | iter2 strict (std) |
|---|---|---|---|
| weak | 57.7% (0.13) | 59.2% (0.24) | 59.6% (**0.26**) |
| medium | 63.6% (0.13) | 63.0% (0.23) | 65.1% (0.24) |
| strong | 88.9% (0.09) | 88.2% (0.11) | 87.9% (**0.15**) |
| strong_eac_all | 24.4 (5.8) | 30.8 (12.3) | 27.9 (**14.5**) |

iter1 → iter2(soft)에서 std가 거의 2배가 됐고, iter2(strict)에서 **strong/EAC의 다이내믹 레인지가 더 늘었다** — 대칭이 봇의 변별력을 *낮추지 않음*.

#### 회귀 R² 비교 (n = 100)

| 종속변수 | iter1 | iter2 soft | iter2 strict |
|---|---|---|---|
| weak_clear_rate | 0.223 | **0.443** | 0.298 |
| medium_clear_rate | 0.287 | **0.464** | 0.302 |
| strong_clear_rate | 0.239 | 0.222 | 0.127 |
| strong_eac_all | 0.287 | **0.322** | 0.221 |

soft → strict로 가면서 **R²가 거의 모두 떨어졌다**. 시각적 패턴 vs 예측력의 트레이드오프.

#### 왜 strict에서 R²가 떨어졌나

가설: strict mirror가 보드의 *구조적 자유도를 절반(또는 1/4)으로 축소*하면서 6차원 벡터 분포가 좁아진다. 봇 성능 분산은 오히려 늘어났는데(strong_eac std 12.3 → 14.5), 이는 6차원이 *캡처하지 못하는 새로운 변동 요인*이 들어왔다는 뜻 — 가장 유력한 후보는 **봇이 좌우(또는 상하) 어느 쪽에서 연쇄를 시작하느냐**라는 *전략적 비대칭성*. 같은 형태의 보드라도 봇의 색 시퀀스에 따라 한쪽 구역이 먼저 무너지면 나머지는 단절되어 어려워진다.

이 현상은 라이브 게임에서도 흔하다 — *대칭 보드는 봇 측정에 노이즈가 더 들어간다*. 다음 iter에서:
- calibration_runs 60 → 200으로 늘려 봇 시퀀스 노이즈 평균화
- 또는 봇 결정에 좌우 대칭 헤이스틱 추가 (양쪽을 비교 후 한 쪽만 작업)

#### 다중공선성 — 미세 추가 완화

| | iter1 | iter2 soft | iter2 strict |
|---|---|---|---|
| corr(density, shot_pressure) | **-0.976** | -0.724 | **-0.676** |

#### 난이도 곡선 검증 (iter2 strict)

| | corr |
|---|---|
| level_index ↔ weak_clear_rate | **-0.748** |
| level_index ↔ strong_eac_all | **+0.535** |

soft와 비슷한 강도 — 색 진행 공식이 strict에서도 그대로 동작.

#### Strong 천장 효과 — 부분 해소

strict에서 strong_clear_rate의 std가 0.11 → **0.15**로 늘었고, 최솟값이 0.50 → **0.32**까지 떨어짐. 어려운 strict 레벨에선 strong 봇도 진짜 고전.

### 요약: 세 배치의 진화

| | 수익 | 비용 |
|---|---|---|
| iter1 → iter2 (soft) | R² 두 배, 다중공선성 완화, 난이도 곡선 작동 | 대칭이 너무 부드러워 *패턴으로 안 보임* |
| iter2 (soft) → iter2 (strict) | **시각적 패턴 완성**, EAC 다이내믹 레인지 더 확장, strong 천장 효과 일부 해소 | R² 0.44 → 0.30 (-0.14). 새 노이즈는 6차원이 캡처 못함 |

이 트레이드오프 자체가 **라이브 게임 디자인의 핵심 갈등** — 미감과 데이터 신호 사이. 두 배치를 보존해 디자이너가 *실제로* 둘을 비교하고 결정할 수 있도록 했다.

## 6.6. Iteration 3 — 게임 물리 준수 + 레벨별 Rationale (ADR 0007)

iter2 strict 배치 검수 중 **L008에서 고립된 부유 셀**이 발견됨. 게임의 가장 기본 규칙 — *"버블은 천장 또는 다른 버블에 연결돼야 한다"* — 위반. 동시에 디자이너 피드백: *"각 레벨에 Why를 자동으로 적어라"*.

### 변경 사항
1. **시드 천장 앵커링**: Stage 1 시드는 row 0에만 배치 → BFS 성장으로 *모든* 셀이 천장과 연결 보장.
2. **자동 부유 제거**: Stage 2 변형 직후 매번 `Board.drop_floating()` 호출. fitness는 *playable 상태*에서 평가.
3. **상하 대칭 폐기**: 정의상 바닥 미러는 천장 비연결 → 모두 부유. 좌우 대칭 또는 비대칭만 허용.
4. **레벨별 Rationale 자동 생성**: `src/bubble/rationale.py`. 룰베이스로 (spec, achieved features, 봇 캘리브레이션) → 7-필드 디자이너 노트.

### 연결성 invariant 검증
새 테스트 `tests/test_connectivity.py` — 3개 테스트 모두 통과 → **모든 생성 보드의 부유 셀 0 보장**.

### Rationale 예시 (smoke output)
```
L003: 초입 (2색)
summary       : 초입 2색 — 큰 콤보 훈련
axis          : 좌우 대칭 패턴 — 시각적 일관성, 양쪽 동시 학습 가능
intent        : 한 발에 7개 이상이 깨지는 클러스터를 의도적으로 배치해,
                플레이어가 큰 콤보의 시각·청각 보상을 경험하도록 만든다.
key_feature   : max_chain_depth
achieved      : max_chain_depth 일치 (목표 7.00 → 도달 7.00)
expected_play : 기초 — 즉각 클리어 가능, 온보딩 적합. 봇 W100% / S100%, 평균 17발.
skill_trained : 큰 클러스터 식별과 한 발 콤보.
```

### 결과 (n = 100, 1시간 35분 배치)

iter2 strict v2 산출물은 `levels_iter2_strict_v2/`, `reports_iter2_strict_v2/`로 보존됨. 비교 가능.

#### 봇 분포

| 봇 | iter2 strict mean (std) | iter3 mean (std) | 변화 |
|---|---|---|---|
| weak | 0.596 (0.257) | 0.618 (0.252) | 평균 +0.02, std 변화 없음 |
| medium | 0.651 (0.244) | 0.660 (0.231) | 평균 +0.01 |
| strong | 0.879 (0.149) | 0.902 (0.117) | 평균 +0.02, std 0.15→0.12 |
| strong_eac_all | 27.86 (14.48) | 26.85 (14.12) | 평균 -1.0 |

부유 셀 제거로 *밀도가 살짝 줄어* 평균 클리어율이 1-2%p 상승. std는 거의 유지.

#### 회귀 R² — 부유 셀이 노이즈였음을 확정

| 종속변수 | iter1 | iter2 soft | iter2 strict | **iter3 (post-fix)** | strict→iter3 Δ |
|---|---|---|---|---|---|
| weak_clear_rate | 0.223 | **0.443** | 0.298 | **0.393** | **+0.10** |
| medium_clear_rate | 0.287 | **0.464** | 0.302 | **0.428** | **+0.13** |
| strong_clear_rate | 0.239 | 0.222 | 0.127 | 0.194 | +0.07 |
| strong_eac_all | 0.287 | 0.322 | 0.221 | 0.251 | +0.03 |

**핵심 발견**: ADR 0007의 가설 — *"부유 셀은 회귀 노이즈"* — 가 정량 확인. 부유 셀을 제거하니 R²이 strict 대비 weak +0.10 / medium +0.13 회복. iter2 soft 수준(0.44/0.46)에는 미달하지만, **시각 패턴 + 게임 물리 + 디자이너 노트**까지 갖춘 완성형 배치.

#### 회귀 계수 (`strong_eac_all`)

| 차원 | iter2 strict | iter3 |
|---|---|---|
| color_entropy | +4.14 | +3.39 |
| shot_pressure | +4.79 | **+9.48** ★ |
| density | +1.97 | +8.88 |
| avg_cluster_size | +1.97 | **-1.70** (부호 정정!) |
| max_chain_depth | +1.35 | +1.16 |
| floating_potential | -2.52 | +0.49 |

iter2 strict에서 *부호가 직관과 어긋났던* `avg_cluster_size`가 iter3에서 **-1.70으로 회복** — 클러스터가 클수록 시도 횟수↓. 부유 셀이 만들어내던 *간접 효과*가 사라진 결과. 이것 하나로도 ADR 0007의 정당성이 입증.

`floating_potential` 계수가 -2.52 → +0.49로 무력화 — 부유 셀 자체를 금지했으니 *측정 가능한 부유*는 *진짜 키스톤 구조에서만* 발생, 그 효과가 회귀에 더 깨끗하게 잡힘.

#### 다중공선성 — 살짝 증가

| | iter1 | soft | strict | iter3 |
|---|---|---|---|---|
| corr(density, shot_pressure) | -0.976 | -0.724 | -0.676 | **-0.780** |

iter3에서 다시 살짝 올라감. 이유: drop_floating으로 *post-physics density*가 spec 타겟에 더 빠르게 수렴 → 그룹 내 분산이 줄어 shots/density 비율이 더 일관 → 두 차원이 다시 비슷해짐. 다음 iter의 후보: shots를 *density 그룹마다* 변량화 (현재는 num_colors 그룹마다 변량).

#### 난이도 곡선 검증 (iter3)

| | corr |
|---|---|
| level_index ↔ weak_clear_rate | -0.681 |
| level_index ↔ strong_eac_all | +0.490 |

soft/strict와 같은 강도. 색 진행 공식이 일관되게 작동.

#### 연결성 invariant — 부유 셀 0개 보장

`tests/test_connectivity.py` 3개 테스트 통과 + `Board.drop_floating()` 모든 변형 후 호출 → 100레벨 모두 *playable 상태*.

### 요약: 4개 배치의 진화

| 배치 | 시각 패턴 | 게임 물리 | weak R² | medium R² | 디자이너 노트 |
|---|---|---|---|---|---|
| iter1 | ❌ | ❌ (부유 가능) | 0.22 | 0.29 | ❌ |
| iter2 soft | △ | ❌ | **0.44** | **0.46** | ❌ |
| iter2 strict | ✅ | ❌ | 0.30 | 0.30 | ❌ |
| **iter3** | ✅ | **✅** | 0.39 | 0.43 | **✅** |

iter3는 *완성도 + 신뢰도* 둘 다 갖춘 첫 배치. R²은 iter2 soft 대비 약간 낮지만, **물리 정합성·디자이너 노트·시각 패턴**의 가치가 그 0.05를 정당화한다.

## 6.7. Rationale v2 — 공식·패턴 가시화 (2026-05-13)

iter3의 *디자이너 노트* 7필드가 *왜 이 레벨이 이런 모양인가*에 답한다면, v2는 그 위에 *어떤 알고리즘이 적용됐는가*를 한 단계 더 박는다. 각 레벨 카드에 적용된 공식·패턴이 자동으로 따라붙어, 후임 디자이너나 데이터 분석가가 코드를 열지 않고도 추적할 수 있는 구조.

### 동기

iter3의 부유 셀 결함 추적은 *코드를 따라가야* 가능했다. *어떤 알고리즘이 어디서 결함을 만들었는지*를 매 레벨 카드에 박아뒀다면 추적 시간이 줄었을 것. 라이브 운영에서 *디자인 의도 + 알고리즘 적용 이력* 두 층을 같이 본다.

### 구조

각 레벨 JSON의 `rationale`에 두 필드 추가:
- **`formulas`** — 수학적 계산식. *공식·값·기능·왜* 4필드
- **`patterns`** — 알고리즘적 구조. *효과·왜* 2필드

자동 추출 항목:

| 카테고리 | 항목 | 적용 조건 |
|---|---|---|
| 공식 | `band_classification` (rationale `_band`) | 항상 |
| 공식 | `color_progression` (ADR 0006) | 항상 |
| 공식 | `shot_pressure_formula` (ADR 0008) | 항상 |
| 공식 | `weighted_distance_fitness` (ADR 0004) | spec.weights 존재 |
| 공식 | `target_sampling_ranges` (sample_specs) | 항상 |
| 패턴 | `two_stage_generation` (ADR 0004) | 항상 |
| 패턴 | `ceiling_anchor` (ADR 0007) | 항상 (iter3+) |
| 패턴 | `drop_floating` (ADR 0007) | 항상 (iter3+) |
| 패턴 | `horizontal_mirror` (ADR 0006) | 항상 (2026-05-13 결정 이후) |
| 패턴 | `bot_triplet_calibration` (ADR 0005) | 항상 |

*왜*는 ADR(0004·0005·0006·0007·0008)에서 그대로 인용 — 시스템의 결정 근거가 매 레벨 카드에 박힌다.

**비대칭 패턴 제거 (2026-05-13)**: 사용자 결정으로 `sample_specs`의 25% 비대칭 분기 제거. 모든 신규 레벨이 좌우 대칭으로 강제 생성됨. 라이브 상용 퍼즐(Bubble Pop Origin·Royal Match)의 표준 정렬에 맞춘 결정이며, iter1~iter3에서 비대칭이 회귀 신뢰도를 명확히 끌어올린 증거는 없었음.

### 시각화

`viewer/index.html` 좌측 패널의 *디자인 의도* 다음에 두 섹션이 따로 표시:
- *적용된 공식* — 파란 좌측 보더, 공식을 등호 형태로 표시
- *적용된 패턴* — 녹색 좌측 보더

### 문서화 산출물

## 7. 회고

### 잘 된 것
- **3층위 분리**(생성기 / 봇 솔버 / LLM 보조)가 두 번의 이터레이션에서 견뎠다. iter2의 봇 인터페이스 변경이 다른 층을 깨지 않음.
- **`simulate_shot`을 보드 primitive로 두고 features와 solver가 공유**한 결정이 후반에 배당으로 돌아옴 — 매칭/연쇄 규칙을 한 곳만 고치면 전부 일관.
- **봇 트리플렛 + clear_rate spread**를 신호로 쓴 것: 단일 EAC 대비 *어려운 레벨까지 측정 가능*하고, weak/strong 격차로 난이도 상한을 변별.
- **(1+1) 진화의 적합도가 1000× 감소**하는 것을 첫 실험에서 확인 — 절차적 생성이 디자이너 의도에 *수치로* 맞춰감을 정량 검증.
- **15→100레벨에서 R²가 무너진 이유를 추적**해 ADR 0005로 기록. 이 추적 자체가 포트폴리오의 가치.
- **Iteration 2가 가설(다중공선성 + 다이내믹 레인지)을 동시에 해소**해 R²가 두 배 — *디자이너 피드백 기반의 시스템 진화*가 코드와 데이터에서 입증됨.
- **Soft → Strict 대칭 전환의 트레이드오프를 정량화**. 시각 미감은 완벽해졌지만 R²가 0.14 떨어짐. *디자인 결정에 대한 비용을 데이터로 증명*하는 라이브 운영 패턴.
- **부유 셀 발견 → 추적 → 수정 사이클 (iter3)** 이 라이브 게임 운영의 *완전한 결함 회복 사이클*을 시연. R²이 iter2 strict 대비 +0.10~0.13 회복, `avg_cluster_size` 계수 부호가 정상화 — *물리 정합성이 회귀 신뢰도까지 끌어올림*.
- **레벨별 Rationale을 룰베이스로 자동 생성**해 100개 레벨 각각에 디자이너 노트가 붙음. 룰베이스라 *재현 가능 + 감사 가능*. LLM 업그레이드 훅도 코드에 명시 — 미래 작업의 토대.

### 아쉬운 것
- **다중공선성을 코드에 넣기 전에 잡지 못함** (iter1). ADR 0003의 "shot_pressure는 게임 규칙 의존적"이라는 정당화가 *실제 셋업(shots 고정)*과 어긋났다. iter2에서 shots를 변동시켜 완화. 다음엔 *생성 전 합성 데이터*에 회귀를 먼저 돌려 차원의 독립성을 검증할 것.
- **StrongBot 천장 효과는 두 이터레이션 모두 미해결**. 평균 88~89% 클리어. iter3에선 LOOKAHEAD_K를 줄이거나 shots를 더 타이트하게.
- **`avg_cluster_size`의 부호 반전 (iter1 -2.77 → iter2 +1.97)**: 색 수와 클러스터 크기의 음의 상관이 회귀에 *간접 효과*로 들어감. 통제변수 `num_colors`/`level_index`를 회귀에 명시적으로 추가해야 분리됨.
- **Strict 대칭의 R² 손실 원인을 6차원이 잡지 못함** — 봇의 *전략적 비대칭성* (어느 쪽에서 연쇄 시작?)이 새 분산 원천. 차원 추가(*"가장 큰 클러스터의 위치 분산"*) 또는 calibration_runs 증대로 다음 iter에 손볼 여지.
- **iter3에서 다중공선성이 다시 살짝 올라감** (-0.676 → -0.780). drop_floating이 density를 spec 타겟에 더 단단히 수렴시키면서 shots/density 비가 일관되어진 부작용. 다음 iter엔 shots를 *density 그룹별로* 변량화해 디커플링.
- **LLM 보조를 진짜 Claude API로 끝까지 돌려보지 못함**. 모킹 모드까지만 데모. 키 주입 시 동일 인터페이스로 즉시 동작하지만, 실제 API 응답의 *변형 의도(rationale)*를 한 번 더 검증해야 신뢰도 ↑.

### 라이브 게임에 적용한다면
1. **봇 EAC를 실제 유저 평균 시도로 교체**. 우리 회귀 프레임은 EAC 자리만 바꾸면 그대로 굴러간다 — *프로토타입이 프로덕션 분석 프레임*.
2. **A/B 테스트 자동 생성**: (1+1) 진화의 적합도에 "기존 레벨과의 거리" 항을 추가, 세 변형을 같은 목표로 진화 → 라이브 A/B/C 후보로 투입.
3. **LLM 보조의 컨텍스트에 회귀 계수를 주입** (이미 인터페이스 마련됨). LLM이 "density가 가장 강한 음 영향이라고 알려졌으니 거기를 건드리겠다"는 식의 합리적 변형 의도를 내도록.
4. **팀 디자이너의 손쉬운 사용 경험**: CLI 대신 *목표 슬라이더*가 있는 작은 웹 앱으로 묶어 배포.
5. **Rationale을 운영 도구의 일부로**: 라이브에 올라간 레벨에 자동 생성된 Rationale을 QA·기획·CS팀 모두에게 노출. 클리어율 이상치가 발견되면 Rationale로 *원래 의도*와 비교 → 디자인 의도와 실제 유저 행동의 격차를 추적.

### 메타 회고 — Claude와의 협업

이 포트폴리오 자체가 AI 협업 실험이었다. 의도적으로 채택한 규칙:
- **CLAUDE.md에 페르소나·검증·문서화 의무를 못박음** → 매 응답이 일관됨.
- **모든 큰 결정을 ADR로** → 결정 시점의 사고가 코드와 함께 살아남음.
- **선택지 제시 형식 강제 (이유·강점·약점·추천도)** → 사용자가 *결정의 의미를 항상 알고* 진행.
- **실패와 되돌림을 숨기지 않음** → ADR 0005가 그 결과.
- **톤 룰 (2026-05-11 도입)** → AI tells·꾸밈 회피 체크리스트로 모든 문서 자가 검사. 인위적 강조 따옴표·추상 부사·모호 어미·3박자 나열 강박 등 차단 패턴 7개와 강한 형용사·동어 반복·추상 비유 절제 룰.

이 협업 방식의 핵심은 *AI에게 결정을 위임하지 않는 것*. 모든 ⭐⭐⭐ 추천에는 사용자의 명시적 동의가 있었다. AI는 *옵션 분석가*였고, 디자이너는 사람이었다.
